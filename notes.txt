2026-05-05 — Algorithms to consider for cal-imaging ensemble extraction
================================================================

Setup constraint: open-world VR, no trial structure (objects come and
go continuously). This rules out trial-structured methods like TCA,
LFADS-as-typically-applied, GPFA-on-trials.

Already in the script:
- PCA
- NMF (frobenius; l1_ratio = 0, 0.5, 1.0)
- SparsePCA / MiniBatchSparsePCA
- KL-NMF (added 2026-05; uses sklearn 'mu' solver)

Ruled out from past experience:
- ICA / PCA→ICA (Lopes-dos-Santos style cell assemblies). Tested
  previously, was inferior to plain NMF on this data.

Queued to explore (in priority order):

1. Stability across seeds (no package needed).
   For each method+k, run NMF ~10 times with different random_state.
   For each component, find its best match in each other run via
   Hungarian assignment + correlation. Plot mean component-stability
   vs k. Stable components = real ensembles; unstable = noise.
   Right thing to do once plain-vs-KL is decided. Tells you both the
   right component count AND which ensembles to trust.
   Implementation: sklearn + scipy.optimize.linear_sum_assignment.

2. seqNMF (Mackevicius et al. 2019).
   NMF variant that finds sequential temporal patterns rather than
   just synchronously co-active neurons. Highly relevant for VR
   because navigation, place codes, and reward-anticipation produce
   sequences that vanilla NMF misses (NMF treats time bins as
   exchangeable).
   Install: pip install seqnmf

3. Consensus NMF / cNMF (Kotliar et al. 2019).
   Multi-seed NMF + clustering; gives per-component stability scores
   for free. This is essentially the packaged version of item #1.
   Install: pip install cnmf-py

4. CEBRA (Schneider et al. 2023).
   Contrastive learning that aligns the latent embedding to a
   behavioral variable (reward, position, velocity). Since lr_data
   and mov_data are already aligned to the imaging frames, behavior-
   aware embedding is feasible without extra data wrangling.
   Install: pip install cebra

5. dPCA (Kobak et al. 2016).
   Demixed PCA. Even without trial structure, can decompose
   population variance into components attributable to continuous
   task variables (reward state, locomotion, object presence).
   Install: pip install dPCA

Methodological additions worth doing regardless of which method wins:
- Cross-validated reconstruction error. Current rel_error is in-
  sample, so high-component methods will look artificially best.
- Shuffled null using existing f_circshift_rates to determine the
  real dimensionality.
- Marchenko-Pastur threshold for PCA — random-matrix-theory bound
  on noise eigenvalues, gives a hard upper bound on real
  dimensionality.

Component-count sweep currently set to [5, 10, 20, 30, 50]. Capped
at 50 because:
  (a) KL with mu solver is slow at high k,
  (b) right answer for ensemble extraction usually lives in 10-50,
  (c) n_neurons / 4 to n_neurons / 2 is the useful upper bound.

2026-05-05 — Why ridge / elastic-net / lasso NMF look ~identical at alpha_W = 0.0001
================================================================

Past observation: with l1_ratio = 0 vs 0.5 vs 1.0 and alpha_W = 0.0001,
the three NMF variants give very similar results. Why:

1. alpha_W = 0.0001 is tiny relative to the reconstruction loss.
   The penalty term sits several orders of magnitude below ||X - WH||²,
   so the l1/l2 split barely affects the optimum.

2. NMF on cal-imaging data is already sparse-ish by construction:
   - non-negativity constraint biases toward parts-based factors,
   - nndsvda init produces sparse W and H,
   - most neurons are silent most of the time after smoothing.
   The marginal sparsity gain from L1 over L2 is small.

To actually exercise the regularization, sweep alpha_W over
[0, 0.001, 0.01, 0.1] with l1_ratio = 1 (lasso). Components will get
progressively sparser. Run this AFTER plain-vs-KL settles, on the
winning beta_loss; otherwise it's premature.

2026-05-05 — KL divergence vs Frobenius loss in NMF
================================================================

Core difference is the implicit noise model:

- Frobenius (||X - WH||²): assumes additive Gaussian noise with
  constant variance. Every error counts the same regardless of where
  in the data it happens.
- KL divergence (Σ X log(X/WH) - X + WH): assumes Poisson noise
  where variance scales with mean. Errors at high-rate points are
  weighted differently than errors at silent points.

Why KL is theoretically better-suited for spike rates:

1. Variance-mean coupling is real in the data. A neuron firing at
   rate 10 has higher variance than one at 0.5 (Poisson: σ² = μ).
   Frobenius treats both errors equally and wastes component mass
   trying to fit noise on silent neurons. KL automatically
   downweights low-rate errors.

2. Most of the data is small numbers. Cal imaging spike rates are
   mostly near zero. Frobenius is dominated by squared-error
   penalties → a few bright cells in a few frames dominate the loss.
   KL penalizes proportionally, giving the dim majority a fairer say.

3. KL produces sparser, more interpretable components. The loss is
   asymmetric — under-estimating a high value costs more than over-
   estimating a low value. Components fit active periods cleanly
   without smearing mass onto silent regions.

4. Multiplicative updates preserve non-negativity for free. KL's
   gradient drives sklearn's 'mu' solver. W and H stay strictly ≥ 0
   throughout; no correction needed.

5. Better matched to the generative process. Spike events per frame
   are approximately Poisson. KL's loss IS the Poisson log-likelihood
   (up to constants) — when data is Poisson-ish, KL is doing
   maximum-likelihood estimation.

Trade-offs (where Frobenius still wins):

- Speed. The mu solver is much slower than cd, typically 3-10× per
  iteration, and often needs more iterations. Painful at high k.
- Numerical fragility. log(X/WH) diverges if WH approaches zero
  where X > 0. sklearn handles with epsilon clipping but can slow
  convergence.
- Cleaner theoretical guarantees. Frobenius NMF has richer
  convergence/identifiability theory. KL has these but less mature.
- Frobenius is correct when noise truly is Gaussian. E.g., raw or
  denoised fluorescence traces (not spike rates) — frobenius is fine
  there. The Poisson story only applies when X represents counts/rates.

Practical analogy:
  Frobenius NMF ≈ least-squares regression
  KL NMF ≈ Poisson regression

2026-05-05 — KL / JSD as a similarity matrix (vs cosine)
================================================================

Question: can KL be used to build similarity matrices like cosine is
used at VR_ca_dimred.py:81?

Yes, but two issues with raw KL:

1. KL is not symmetric: KL(P||Q) ≠ KL(Q||P). Most downstream tools
   (hclust, MDS, UMAP) assume a symmetric distance matrix.
2. KL is defined on probability distributions. Both inputs need to
   be non-negative AND each row should sum to 1 (or you get weird
   scale-dependent behavior). For firing rates: L1-normalize each row.

Symmetric variants you'd actually use:

- Jensen-Shannon divergence (JSD). Most common.
    from scipy.spatial.distance import jensenshannon
    # Returns sqrt(JSD), which is a true metric.
    # Symmetric, bounded in [0, sqrt(log 2)], handles zeros.

  Full pairwise matrix:
    from scipy.spatial.distance import pdist, squareform
    P = X / X.sum(axis=1, keepdims=True)   # rows sum to 1
    dist_mat = squareform(pdist(P, metric='jensenshannon'))

- Symmetric KL: (KL(P||Q) + KL(Q||P)) / 2. Less common, less well-
  behaved (still unbounded, blows up when supports differ).

- Bhattacharyya coefficient: BC(P,Q) = Σ √(P_i × Q_i). Closer to
  cosine in spirit — bounded [0, 1], symmetric. Use 1 - BC as distance.

Cosine vs JSD comparison for clustering neurons:

| Aspect          | Cosine                       | JSD                              |
|-----------------|------------------------------|----------------------------------|
| Bounded         | [-1, 1]                      | [0, sqrt(log 2)] (bits)          |
| Symmetric       | Yes                          | Yes                              |
| Handles zeros   | Fine                         | Fine (with sqrt form, base=2)    |
| Scale-invariant | Yes (purely angular)         | Requires sum-1 normalization     |
| Interpretation  | Angular alignment            | Distinguishability of distrs.    |
| Speed           | Very fast                    | Slower (log + sum per pair)      |

JSD is the principled choice when you're already in a KL/Poisson
framework downstream (e.g., KL-NMF). For neural firing patterns,
cosine and JSD usually give very similar clustering when the data is
clean. JSD is more principled when:
- Neuron rates have very different magnitudes (cosine washes that
  out, JSD respects it).
- You want an information-theoretic interpretation (JSD is in bits,
  the average bits to distinguish the two distributions).
- The downstream method (KL-NMF) shares the probability-distribution
  view.

Most likely it won't flip the answer for ensemble extraction — but
worth A/B once KL-NMF results land, just to see if hclust ordering
shifts.

2026-05-06 — JSD vs cosine: when each one wins (deeper dive)
================================================================

Honest summary: JSD is NOT strictly better than cosine. They measure
different things and which one is more useful depends on what you
want to detect.

What each measures:
- Cosine: "do these neurons fire in the same temporal pattern,
  regardless of how active each one is?" Geometric — angle between
  vectors. Scale-invariant; doesn't care about absolute rates.
- JSD: "do these neurons have the same temporal probability
  distribution?" Information-theoretic — bits of "extra surprise" if
  you used one neuron's firing-time distribution to predict the
  other's. Requires valid probability distributions.

After max- or L1-normalization the two converge a lot, but they
diverge in specific ways.

Cases where JSD does something cosine doesn't:

1. Probability-mass weighting. Cosine treats all time bins equally —
   a 0.01 vs 0.001 difference in a silent bin counts the same as
   0.5 vs 0.4 in an active bin. JSD weighs by probability mass, so
   differences in active bins matter more. For sparse cal-imaging
   data, JSD focuses on the bins that actually carry information.

2. Information-theoretic interpretation. JSD is in bits — JSD = 0.3
   means "0.3 bits on average to distinguish samples drawn from P
   vs Q". Cosine has no scalar interpretation like that.

3. Distribution-shape changes. Bursty vs sustained firing of the
   same total mass — cosine on max-normalized rates can be
   misleading because the bursty one dominates a single bin. JSD's
   probability-distribution view makes the comparison more honest
   about WHERE the mass lives.

4. Consistency with downstream KL-NMF. If KL-NMF wins, using JSD
   upstream closes a small conceptual gap (both treat firing as
   probability distributions instead of geometric vectors).

Cases where cosine wins:

1. Speed. Cosine is much faster — matters for large N.

2. Anti-correlation detection. Cosine returns [-1, 1], so reciprocal
   firing patterns get similarity = -1. JSD only measures
   dissimilarity (≥ 0), no notion of "opposite" — reciprocal and
   random patterns just both look "very different".

3. Negative inputs. Cosine works on z-scored data or residuals. JSD
   requires non-negative probability distributions, full stop.

4. Established in cal-imaging literature. Cosine is the standard for
   cell-assembly detection in rodent ephys/imaging papers — needs
   no defending to reviewers.

Practical take for our data:

JSD is more principled IF we're in a KL framework downstream (i.e.
KL-NMF). On max-normalized spike rates, the practical difference for
hclust ordering is often modest — tree moves a little, heatmap looks
slightly different, rarely dramatic.

Strongest reason to use JSD: conceptual consistency with KL-NMF. If
KL-NMF wins the benchmark and becomes the method of choice, JSD for
the upstream hclust closes a small theoretical gap. If we stay with
frobenius NMF, sticking with cosine is reasonable (faster, more
standard).

Decision: A/B both metrics with the same data. If they agree on
ordering, use cosine (faster, more standard). If JSD finds structure
cosine missed AND that structure is biologically interpretable,
that's the case for switching.

2026-05-18 — Decoding diagnostics: negative R² and how to read it
================================================================

First CEBRA run, blocked-CV KNN(k=15) decoding from 3D embedding:
  PCA-3 (unsup):  R² < 0 on both agg and pix targets
  CEBRA-agg:      R² < 0 on both
  CEBRA-pix:      R² < 0 on both, but pix > agg

Note: pix-PCA features beat agg features as a SUPERVISION signal for
CEBRA. Likely because:
  - pix renders the actual visual scene (so distance → angular size is
    baked in via projection); agg passes raw distance.
  - pix carries spatial structure of multiple objects; agg only tracks
    the nearest object per side.
This is why we added agg_mode = 'angular' / 'salience' / 'angular_salience'
gating in the build cell — to test whether closing the geometric gap on
the agg side narrows the supervision difference.

Why R² is going negative — root cause is NOT just that CEBRA failed:

The decoding cell predicts the FULL target vector at once (8 agg cols
or 20 pix-PCA cols) from a 3D embedding, then reports
multioutput='uniform_average' over columns. R² per column is
  R²_c = 1 - SS_res_c / SS_tot_c       (where SS_tot uses TEST fold mean)
Several target columns have tiny SS_tot:
  - presence_L/R: binary, mostly 0 → low variance
  - vert: mice rarely look up/down → low variance
  - pix-PCA components 10-20: little visual variance, just noise
For these columns, even small errors give R²_c well below 0 (e.g. -10).
The uniform average then drags the reported number negative even when
lat and dist are actually decoded reasonably.

PCA-3 baseline going negative is for the SAME reason — predicting 8 or
20 simultaneously from 3 unsupervised dims is over-specified. The
baseline isn't "broken," it's reporting the right thing about an
under-specified question.

Three loosening directions (queued as tasks):

(1) PCA decoding sweep — STARTING NEXT.
    Vary n_neural_PCs as input embedding (1..50). Decode each behavior
    feature INDIVIDUALLY (8/10 agg cols + 20 pix-PCA cols, 28-30 curves
    total). Per-column R² as function of n_pcs. Heatmap + line plots.
    Tells us which features are in neural variance at all and how many
    dims each needs. Replaces the misleading single-number decoding.

(2) Per-feature R² for CEBRA, plus variance_weighted aggregation.
    Same per-column decomposition for CEBRA embeddings. Adds
    multioutput='variance_weighted' as an alternative aggregation that
    down-weights low-variance columns (so presence doesn't dominate).

(3) Restrict-targets ablation.
    Drop presence and vert from targets — keep just [lat_L, dist_L,
    lat_R, dist_R]. See if uniform_average becomes positive. Tests
    whether the negative-R² is a column problem rather than an
    embedding problem.

Methodological note: KNN(k=15) on a 3D embedding may be too few
neighbors when n_pcs gets larger (curse of dimensionality). At n_pcs=50
the local neighborhood is sparse and k=15 may overfit. Worth re-running
the chosen n_pcs at multiple k values to be safe.

2026-05-18 — Adversarial nonstationarity in top pix-PCs (OPEN ISSUE)
================================================================

Issue: pix-PC1 and pix-PC2 real R² falls FAR below shuffled-control null
in the PCA decoding sweep (VR_ca_cebra.py — shuffle cell). Bottom row
of the per-feature Δ-R² plot shows real << null for those PCs.
Task #8 tracks this with sub-tasks #9, #10, #11.

What it's not:
 - NOT "feature isn't in neural representation" — that pattern would
   give real ≈ null, both near zero. Real being WAY below null means
   the real decoder is actively misled, not just empty.
 - NOT a CV blunder — the same code does the right thing for other
   features. Only the slow-target ones break.

Diagnosis (likely cause): adversarial nonstationarity.

Why top pix-PCs specifically:
 PC1 of the rendered pixel movie tends to be a slow, session-spanning
 signal — e.g. "is any object on either monitor at all" (overall
 brightness) or "left-monitor vs right-monitor balance". Both change
 on minute timescales, not frame-to-frame. PC2 is similar (next slowest
 mode). These are the slowest, most nonstationary features in the
 target set.

Mechanism in blocked CV:
 1. The decoder is trained on the first 4 of 5 contiguous time blocks.
    The pix-PC1 trajectory across those blocks has some pattern,
    say monotonically increasing.
 2. Neural drift also exists — global arousal, baseline shifts, etc.
    During training, neural PC1 and pix-PC1 may correlate spuriously.
 3. The decoder learns that correlation as a direction.
 4. In the held-out block, the relationship has drifted — different
    neural baseline, different visual content distribution. The
    decoder applies the learned direction in the WRONG sign or scale.
 5. Predictions point opposite the truth → R² strongly negative.

Why shuffling rescues it:
 - circshift_target rolls the behavior by ~10 s. The slow structure
   in pix-PC1 is largely preserved (a 10 s roll doesn't change minute-
   timescale trends much).
 - But the spurious training-block alignment with neural is destroyed.
   The decoder, trained on no consistent relationship, just learns to
   predict near the mean → R² ≈ 0 (mildly negative).
 - So null R² > real R² for these specific PCs, not because the null
   is good, but because the real is actively bad.

Confirmation diagnostics queued (task #9):
 - Plot X_pix[:, 0] and X_pix[:, 1] vs frame index — look for slow
   drift across the session.
 - Report per-fold R² (not just the 5-fold mean) for pix_PC1 and
   pix_PC2. Adversarial pattern: one fold ~ +0.3, others ~ -3.
   Null pattern: all folds near 0.

Fixes queued:
 - Task #10: detrend the targets. Subtract a Gaussian-smoothed
   (σ ≈ 5-10 s) version of each target column from itself before
   decoding. Removes the slow drift; decoder then learns fast
   structure that generalizes across time blocks. Apply to both
   X_agg and X_pix.
 - Task #11: optionally also detrend the neural embedding columns.
   Smaller expected effect; try if target detrend alone isn't enough.

Why this matters methodologically:
 If we don't fix this, the per-feature Δ R² plot is misleading for
 the slow features — it would say "neural population doesn't encode
 pix-PC1" when actually it might encode it, the blocked-CV decoder
 just can't extract it without detrending. The whole point of having
 a shuffled control was to give the right baseline; the control is
 working correctly here, the DECODER is misbehaving on slow targets.

Code reference: VR_ca_cebra.py, sections:
 - PCA decoding sweep (lines ~510-650)
 - Shuffled control (lines ~660-800)

2026-05-19 — VR camera convention for this rig
================================================================
The camera in Unity is placed at exactly 0.4 world units above the
terrain at the mouse's (x, z). mov_data['y_pos'] records this camera
y (not body base, not a fixed altitude), so it varies as the mouse
moves over terrain.

Spawn behavior: at session start, the mouse spawns at y=15 (above
terrain) and falls under Unity physics until it settles at
terrain_y + 0.4. The first few frames (~0.5 s of real time, depends
on fall distance) report inflated y_pos values and should be treated
as transients for any downstream analysis.

Implication for f_render_terrain:
 - eye_height = 0 is correct — no offset needed; the renderer uses
   mouse_xyz as-is. Camera ends up at terrain + 0.4 after settling.
 - The mouse-vs-terrain alignment histogram peaks at ≈ 0.4 with some
   spread from NN-lookup quantization on the discrete terrain grid;
   a long tail toward larger delta values comes from spawn-and-fall
   frames.

2026-05-19 — h/y swap bug in f_angles_to_movie_v2 vertical edges
================================================================

Discovered while debugging "object below surface of terrain" after
adding Z-buffer compositing. Despite the compositing being correct,
parts of objects rendered underground were still visible.

The renderer's edge formulas used obj_size['y'] and obj_size['height']
on the WRONG sides of the silhouette. Conventions:

  - obj_size['y']      = object half-extent above its registered center.
  - obj_size['height'] = offset of the registered center above the ground
                         (and thus the angular extent below the center
                         that should be visible — the rest is underground).

In pixel space, `vert_pix = (vert_angle + vFOV/2)/vFOV * (vert_samp-1)`,
so positive vert_angle (object above camera) maps to LARGER pix index;
larger pix = higher in world. The mask should therefore cover:

  lower_pix (= bottom of object in world) = vert_idx - obj_h_ang_pix
  upper_pix (= top    of object in world) = vert_idx + obj_y_ang_pix

Pre-fix (v1 f_angles_to_movie AND v2 f_angles_to_movie_v2):

  edge_top    = vert_idx - obj_y_ang_pix   # WRONG — applied y to the bottom
  edge_bottom = vert_idx + obj_h_ang_pix   # WRONG — applied h to the top

Net effect: the rendered silhouette extended the full obj_size['y']
BELOW the center (into the underground), while the top was capped at
only obj_size['height'] above center. The whole painted region sat
~(y - height)/2 lower in screen-vertical than intended.

The bug was invisible in clip_underground=False mode because that path
uses obj_y_ang_pix on both sides (symmetric ±y around center) — the
swap doesn't change anything when the two values are equal.

Fixed in v2 only (both antialias and non-antialias paths). Legacy v1
f_angles_to_movie left alone as the historical reference.

How to verify: re-render with terrain + Z-buffer composite; the
object's bottom should now sit at the terrain horizon at its xz
location, and the top should reach center + obj_size['y']. With
obj_size['height']=2 and obj_size['y']>2, the visible silhouette is
taller post-fix than pre-fix.

2026-05-20 — vertical orientation fix: negate vert_angle in pixel mapping
================================================================

Symptom: in default ImageJ / matplotlib display (row 0 at top of
screen) the rendered monitor frames were vertically inverted —
terrain appeared at the top half of the screen and the sky / cylinder
tops at the bottom. Confirmed visually with orientation debug stripes:
a bright row-2 stripe sat at the top of the display (as expected),
but the world's "above camera" content was rendering at large pix =
bottom of the display.

Root cause: the lateral pixel formula was already negated
(`-vec_data['obj_lat_angle']`) so that positive lat_angle maps to the
right column of the display, but the vertical formula was NOT
negated. Positive vert_angle (object above camera) was mapping to
larger pix = bottom of the display.

2026-05-20 — terrain cell_size correction: 3-cell overlap, cs = 1.0
================================================================

Earlier convention (now corrected): cell_size derived as
chunk_pitch / cell_max = 122 / 124 ≈ 0.984. This assumed 1-cell
overlap between adjacent chunks.

Direct check: pivoted two x-adjacent chunks (A at ChunkPosX=-610 and
B at -488, pitch=122) and compared boundary rows. The RMS-diff
between A's last 3 rows and B's first 3 rows formed an anti-diagonal:

              B[x=0]   B[x=1]   B[x=2]
A[x=122]    →  0.0000
A[x=123]              0.0000
A[x=124]                       0.0000

So the actual overlap is **3 cells per side**, not 1. From the
geometry:

  A[124] world_x = A_center + (124 − 62) * cs
  B[2]   world_x = B_center + (  2 − 62) * cs
  ⇒ A_center + 62*cs = B_center − 60*cs
  ⇒ 122*cs = B_center − A_center = chunk_pitch = 122
  ⇒ cs = 1.0

Final convention for this rig (locked in 2026-05-20):
  - cells per chunk: 125 (indices 0..124, cell_max = 124)
  - chunk_pitch (center-to-center): 122 world units
  - cell_size: 1.0 world unit per cell
  - chunk width: cell_max * cell_size = 124 world units
  - overlap per side: 3 cells = 2 world units
  - chunk_centered=True (ChunkPos is at the cell-62 center)
  - flip_z=True

Code change: `f_terrain_world_coords` now takes an explicit `cell_size`
parameter (default 1.0). The chunk_pitch/cell_max derivation is gone.
The same parameter threads through `f_render_terrain` (both stride>=1
and stride<1 paths). `_terr_kw` in VR_ca_cebra.py now passes
cell_size=1.0.

~0.8% — small enough that object-vs-terrain alignment still came out

2026-06-01 — Port of MATLAB ensemble_analysis_YS pipeline to Python
================================================================
Existing track in VR_ca_dimred.py runs an NMF/PCA sweep across k with
in-sample reconstruction error + Hoyer sparsity + Hungarian
component-stability across seeds + circshift-shuffle null on
explained variance. That answers "which method is reconstructing well",
not "which components are real ensembles" or "what is the right
smoothing / k for held-out data". The MATLAB pipeline at
C:\Users\ys2605\Desktop\stuff\ensemble_analysis_YS\ has both:
out-of-sample CV-driven param selection, and threshold-based
cell→ensemble assignment with a shuffle null. We are porting the
relevant pieces.

Four methods ported into functions/f_ensembles.py:

  A. CV grid sweep (smooth_SD x num_comp) using leave-neuron-out test
     error. Each fold smooths + normalizes train, fits NMF/PCA, then
     predicts each held-out cell from the factor basis of the others
     on the raw test fold. Reports test_err / train_err / test_norm
     averaged across folds. Optional shuffled-null grid via
     with_shuffle=True.
       Entry: f_cv_estimate_grid → pandas.DataFrame.

  B. Auto-num-comp via shuffle PCA eigenvalues. Fit PCA on real X +
     n_shuf circshift-shuffled copies; count components whose
     explained-variance ratio exceeds the per-shuffle max. Mean count
     across shuffles → dimensionality_corr (ceil = recommended k).
       Entry: f_estimate_dim_corr.

  C. Threshold-based extraction. Per-component cutoff via signal_z
     (median + 2.5 * MAD-like spread) or shuff (95th percentile of
     50-shuffle NMF coeff/score pool). Assigns cells (coeffs > thresh)
     and trial frames (scores > thresh) to ensembles. Output ens_out
     mirrors MATLAB shape: cells.ens_list[i], cells.clust_ident,
     cells.non_ens_list, ens_scores; same for trials.
       Entry: f_ens_get_thresh + f_apply_thresh.

  D. Cluster-based extraction. Filter cells whose mean pwcorr exceeds
     the shuffle null at the 95th percentile per cell, then hclust on
     coeffs (Euclidean default for NMF, cosine for PCA/ICA). Cells
     below the filter stay labelled 0 ("unassigned"). Output mirrors
     Method C.
       Entry: f_extract_clust.

Top-level orchestrator: f_ensemble_extract(X, num_comp=None,
smooth_sigma_bins=None, dred_method=..., extraction=..., ...). If
num_comp is None, calls Method B to auto-pick. Returns the merged
ens_out dict.

MATLAB-style plots in functions/f_ensemble_plots.py: f_plot_cv_grid

2026-06-06 — Visual-cortex-style monitor input representations (plan)
================================================================
Current monitor features decode weakly (position barely above/at chance; e.g.
ang_size in-sample train R² only ~0.3 even at k=50 PCs, test negative). The two
representations are at opposite wrong extremes vs how visual cortex encodes the
scene:
  - agg  = one object-CENTER point (lat/vert/dist) → no edges, no spatial
           extent, no retinotopic location, only coarse center velocity.
  - pix  = PCA on raw rendered intensity → top PCs are global brightness /
           "anything on screen" / L-R balance (the slow drift modes behind #8).
           Linear PCA on raw pixels is not aligned to any cortical computation.
Cortex is LOCATION-selective and DIRECTION-selective (oriented edges at
retinotopic locations, with motion), and applies nonlinearities (rectify,
square) BEFORE a roughly linear readout. Our decoder is linear (ridge) with a
front-end that has none of that.

Organizing principle (canonical encoding-model pipeline, Hubel-Wiesel ->
Adelson-Bergen motion energy -> Gallant-lab encoding models):
    filter -> nonlinearity -> spatial/temporal pool -> compress -> linear decode
i.e. put the nonlinear visual front-end into the FEATURES. Substrate is ideal:
two_mon_frames is ~101x101 per monitor in ANGULAR coords (pixel ~ degree),
crisp synthetic silhouettes, known geometry.

Options (all become new built_blocks behind build_* knobs; compute on behavior
clock then resample; PCA-compress; decode through f_blocked_cv_r2). Tiered
cheap->expensive:
  Tier 1 'grid'  — retinotopic tiling: per-cell occupancy/contrast/edge energy.
                   Location selectivity; biggest conceptual fix; trivial.
  Tier 2 'flow'  — dense optical flow (Farneback) pooled into the grid: speed,
                   direction sin/cos, divergence(=local looming). Direction-of-
